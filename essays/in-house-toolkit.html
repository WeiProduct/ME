What it takes to ship seventeen apps as one person.

The in-house toolkit, the time-to-store claim decomposed, and what does not reuse.

People ask how a one-person studio gets to seventeen iOS apps in two years. The answer is unglamorous: most of the work is already done before a new app starts.

The first three apps I built at WeiProduct each rebuilt the same five things — auth, in-app purchase, settings, telemetry, a server proxy in front of OpenAI. By app number four it was obvious the studio was a template, not a sequence of products. So I built the template explicitly. Everything below is part of that template.

What I reuse

  • SwiftUI components. Forms, settings rows, paywalls, onboarding screens, empty states. Pre-styled, pre-accessible, drop in. Each new app picks maybe twelve of them and customizes copy.
  • Auth + IAP. One module. Anonymous Apple ID by default, optional email upgrade, subscription IAP wired to App Store Connect. Has not needed a real edit in a year.
  • Telemetry. A thin client that fires a fixed schema of events (launch, paywall view, purchase, retention milestones). Backed by a tiny aggregation endpoint. Enough to answer the only question that matters early: are people coming back?
  • Server proxy template. A hardened serverless function in front of OpenAI. Origin allow-list, model allow-list, input cap, server-side prompt, rate limit. New app: copy the template, change the system prompt, deploy.

The sixty percent, decomposed

The toolkit cuts average new-app time-to-store by roughly sixty percent. It is worth being honest about where that savings comes from, because the percent has no value in the abstract.

Most of it is not in writing code faster. Most of it is in skipping decisions. When the auth flow is already chosen, I don’t spend an afternoon comparing libraries. When the paywall layout is already designed, I don’t open Figma. The toolkit’s leverage is decision elimination — code reuse is the visible part.

The rest of the savings comes from a long tail of App Store Connect work that was painful the first three times: localized metadata, screenshot templates, review notes, IAP setup, signing. These are now a checklist, not a discovery process.

What doesn’t reuse

The toolkit ends where the product begins. Each app’s core loop — the one screen the user opens every day — gets built from scratch. That is on purpose. If I templatize the core loop, every app starts to feel like every other app, and none of them earn habits. The thesis from the other essay applies here too: the invisible AI lives in the core loop, and the core loop has to be designed for the specific user moment.

The boundary is roughly: chrome reuses, the core does not.

Tools matter less than decisions

The temptation, for any technical founder, is to spend more time on the toolkit than on the products it ships. I have done this. Two months sharpening the template produced fewer new apps, not more.

A toolkit improvement only counts if it removes a decision from the next app. Faster code paths, prettier abstractions, more clever architecture — none of these matter if a new app still has to think about auth, telemetry, or the paywall. Once the decisions are gone, stop improving the toolkit and go ship.

The studio works because the decisions worth making — what habit am I addressing, what does the core loop feel like, does the AI stay invisible — are the only ones left.

— Wei Fu · May 2026
